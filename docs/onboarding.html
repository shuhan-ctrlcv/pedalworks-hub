1 · TL;DR

PedalWorks is a made-up bicycle company that exists only to test a knowledge-base pipeline. It is split across 7 git repos: 6 hold the "company's" own documents and code (these get read by the pipeline — they are ingested), and 1 — pedalworks-hub, this repo — holds the answer key for what the pipeline should produce, and is never itself read by the pipeline.

2 · What this is, and why a fake company

A little jargon first, defined once so the rest of the page reads cleanly:

  • Knowledge graph — a set of facts stored as things (called entities, e.g. "Frame") connected by relationships (called edges, e.g. "Frame made_of Aluminium"). It is how the bot answers questions instead of just keyword-matching text.
  • KV engine — the extraction tool (LightRAG-style) that reads a document and automatically pulls out entities and edges to build that graph. "KV" = key-value, how it stores each fact internally.
  • Ground truth — the actual, hand-verified correct answer. Like an answer key for a test: you can't grade a test without one.

Real customer documents don't come with an answer key — nobody can tell you with certainty every fact that should have been extracted from a real supplier contract. So this project invented a company, PedalWorks, and wrote its documents (order records, supplier docs, assembly steps, ledgers…) ourselves. Because we wrote them, we know exactly what facts they contain — which means we can check whether the pipeline extracted them correctly.

Why split it into multiple repos instead of one folder of documents?

  • Cross-source fusion — a real company's knowledge never lives in one place. Its real test is whether the pipeline can correctly link a fact mentioned in one source (e.g. a part defined by Procurement) to a fact mentioned in another (e.g. Production's bill of materials referencing that same part). A single-repo fixture can't test that at all.
  • Organisation — each repo maps to one business area with one clear owner, so it's obvious where a new fact belongs and who's allowed to define it (see the ownership rule below).

The hub is not a step in this chain — it sits to the side and, after a run, compares the resulting graph against the per-area structure.yaml files under eval/ to check the pipeline got the facts right.

4 · The tier idea (0 / 1 / 2)

Every area's facts are organised by tier — how zoomed-in the fact is:

  • TIER 0 the area's single theme node — the big-picture label for the whole area.
  • TIER 1 the concrete things inside that theme.
  • TIER 2 detail describing a tier-1 thing — materials, sub-parts, build steps.

A real example, zoomed in one level at a time, from the procurement area (eval/procurement/structure.yaml):

T0Supply — the theme node for the whole procurement area
T1Frame — a part, supplied by Frames Ltd, unit cost $120
T2Aluminium — the material Frame is made_of
T2Main Tube Set, Head Tube, Welded Joints — Frame's other has_subpart children

Tiers are a viewing convenience, not something the KV engine outputs directly — see the "overlay honesty" note in SCHEMA.md. The structure check only verifies the right entities and edges are present, not that the extractor reproduced our tier numbers.

What's ingested, and what isn't

  • Ingested — everything under a content repo's docs/ and src/ (and similar): Markdown, PDF, PPTX, DOCX, Jupyter notebooks, spreadsheets, and source code. This is the "company's" actual knowledge.
  • Not ingested — the hub itself — pedalworks-hub is never fed to the pipeline at all (see its own README.md). It only grades output, as in the diagram in §1.
  • Not ingested — YAML / JSON — inside the hub, structure.yaml and qa.yaml are machine-readable ground truth, not prose knowledge, so they would never be treated as ingestible documents even if the hub were ingested. More broadly, the pipeline only extracts knowledge from document/prose formats (Markdown, PDF, DOCX, PPTX, notebooks) — YAML/JSON config-shaped files are not knowledge text.
  • Markdown is ingested — every content repo's docs/*.md is a first-class ingested source; it's the main way facts get into the graph.

6 · How it all connects

Two rules decide where a fact lives (from SCHEMA.md):

  1. Single home for every entity — each entity (e.g. "Frame") is defined in exactly one area, its owner. Other areas may reference it by name but never redefine it.
  2. Edges are filed by ownership, not by tier — if both ends of an edge are owned by the same area, the edge lives in that area's structure.yaml (an internal edge). If the two ends are owned by different areas, the edge lives in eval/_cross_source/structure.yaml instead (a cross-source edge) — regardless of which tier it's at.
Worked example — Trail Blazer uses Frame: "Trail Blazer" is a bicycle model, owned by pedalworks-production (it's a tier-2 entity under Production's "Products" group). "Frame" is a part, owned by pedalworks-procurement (Supply). Because the two ends are owned by different areas, this edge cannot live in either area's own file — it is filed in eval/_cross_source/structure.yaml as a cross-source edge, even though it's a tier-2 (fine-detail) fact.

7 · Adding content (Path A / Path B)

Full detail lives in UPDATE-GUIDE.md; the two paths, summarised:

Path A — add to an existing area (the common case)

StepWhat you doFile(s) touched
1Add the source doc/code to the owning content repoe.g. pedalworks-procurement/docs/…
2Add/extend entities in that area's structure file. Search first — reuse an existing name rather than redefining another area's entityeval/<area>/structure.yaml
3Add internal edges (both endpoints in this area)eval/<area>/structure.yaml
4Add cross-source edges (other endpoint owned elsewhere), referencing the other entity by its existing nameeval/_cross_source/structure.yaml
5Add ≥1 question with repo-qualified expected_sources + reference_key_factseval/<area>/qa.yaml (or _cross_source/qa.yaml if it spans repos)
6Update the human-readable map — only if a cross-source edge was addedeval/_cross_source/system-map.md
7Run the verifier (see §9)—

Path B — add a new Tier-0 node / new repo (e.g. Forecasting, Marketing)

StepWhat you doFile(s) touched
1Create the new content repo with its docs/codee.g. new pedalworks-forecasting
2Register the new reporepo-manifest.yaml
3Create the new area's tier-0 node + tier-1/2 entities + internal edges, and its questionsnew eval/<new-area>/structure.yaml, qa.yaml
4Wire it in with cross-source edges and add the node to the mapeval/_cross_source/structure.yaml, system-map.md
5Run the verifier — nothing in existing areas changes; growth is additive—
Edge case — re-homing an entity: moving an entity's ownership to a different area is the one operation that touches two area files at once: remove the definition from the old area, add it to the new one, and re-file any edges whose ownership changed. Expected to be rare.

8 · Letting an LLM do it

Every update follows the same loop (from UPDATE-GUIDE.md), which you can hand straight to an LLM:

  1. Read SCHEMA.md.
  2. Read the target area's structure.yaml / qa.yaml, plus eval/_cross_source/ (structure.yaml, qa.yaml, system-map.md).
  3. Make the edit.
  4. Run the verifier (§9).

A copy-paste prompt template:

I'm adding [describe the new content] to the PedalWorks knowledge-base test fixture.

1. Read pedalworks-hub/SCHEMA.md for the data model and the two ownership rules.
2. Read pedalworks-hub/eval/<area>/structure.yaml, eval/<area>/qa.yaml, and
   eval/_cross_source/{structure.yaml,qa.yaml,system-map.md} for what already exists.
3. Follow UPDATE-GUIDE.md — tell me whether this is Path A (existing area) or Path B
   (new area), and which files you're touching, before editing.
4. Do not invent facts that aren't in the source document I gave you.
5. Do not redefine an entity another area already owns — search first, reuse the name.
6. After editing, run:
   cd pedalworks-hub && PYTHONPATH=. python -m tooling.multirepo.verify
   Show me the diff before you run it.
What the LLM must never do: invent facts that aren't backed by an actual source document, or edit the answer key (eval/) to match what the pipeline produced instead of what the source documents actually say. The hub is graded against the documents — it can't be adjusted to make a run look better.

9 · Verifying your change

The one command (from UPDATE-GUIDE.md / README.md):

cd pedalworks-hub && PYTHONPATH=. /home/nine/anaconda3/envs/road_asset/bin/python -m tooling.multirepo.verify

What it catches (from tooling/multirepo/verify.py):

  • single_ownership no entity is defined in more than one area.
  • referential_integrity every edge's from/to name actually exists somewhere.
  • parent_tier every entity's parent exists and sits at a strictly lower tier.
  • structure_equivalence / qa_equivalence the merged eval/ data still matches the frozen pre-split baseline — a migration-fidelity check.
  • cited_code_facts specific source-code snippets that a QA question cites (e.g. _LOW_STOCK_THRESHOLD = 1 in pedalworks-order-planning/src/planner.py) still exist verbatim.
  • qa_sources_exist every question's expected_sources file actually exists in a content repo.
  • format_probes at least one PDF/DOCX/XLSX/PPTX/notebook still exists somewhere, so format diversity isn't lost.
